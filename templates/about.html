What Is VV-GPT3?

VV-GPT3 is a fully local, zero-cost AI training and inference platform built for macOS. It allows anyone to train their own GPT-2-class language model on custom text data — books, scripts, chat logs, notes — and then have live conversations with the resulting neural network, entirely on their own machine with no internet, no API keys, and no cloud costs.

The system is built on the same decoder-only causal transformer architecture invented by OpenAI for GPT-2 (June 2019), scaled down to run efficiently on consumer Apple Silicon hardware (M1/M2/M3/M4). It is not a wrapper around an existing AI — it trains a brand new neural network from scratch every time, using your data and your compute.

The web application wraps this engine in a professional-grade command center featuring real-time training telemetry, live loss charts, proactive overfitting detection, a persistent inference chat, and a model management hub — all rendered in a custom glassmorphic UI.

The Evolution: V1 → V3

The original application ("Homemade GPT" V1) was built as a minimum-viable exploration into text generation on Windows. It was functional but severely limited by primitive design choices. VV-GPT3 (V3) was rebuilt with modern local deep-learning practices and a safer VV-GPT3 workflow.

Feature Layer V1 "Homemade GPT" V3 "VV-GPT3"
Tokenization Character-level. The engine learned individual letters (a, b, c…). Vocabulary size equaled the number of unique characters (~66). No concept of words. BPE (Byte Pair Encoding) via OpenAI's Tiktoken library. Vocabulary of 50,257 sub-word tokens. Understands word fragments, whole words, and common phrases as single units.
Attention Manual nested Python loops for multi-head self-attention. Extremely slow, high memory overhead. PyTorch SDPA (Scaled Dot-Product Attention) with automatic Flash Attention selection. Fused C++/CUDA kernels run directly in GPU memory.
Inference Full context recomputation for every single character generated. O(n²) per token. KV-Caching. Past attention computations are stored in a tensor buffer, achieving O(1) per-token generation after the initial prompt.
Hardware CPU-only. No GPU acceleration. Training took hours even on small datasets. Apple MPS (Metal Performance Shaders) native for M-series GPU acceleration. Also supports NVIDIA CUDA and CPU fallback.
Training UI Terminal-only output. No real-time monitoring, no loss curves, no checkpointing. Real-time WebSocket telemetry with live Chart.js loss curves, overfit detection, parameter tooltips, persistent logs, and automatic best-model checkpointing.
Model Sizes Single fixed architecture (~150K parameters). 6 configurable presets from Nano (~8M) to Large (~253M parameters with GPT-2 BPE), with adjustable dropout, block size, and batch size.
Platform Windows-only with .bat launcher. macOS-native with .command launcher, .app bundle creator, and Apple Silicon optimization.

The Mathematics of a Transformer

This section explains the exact mathematical operations the neural network performs, from raw text input to generated output. Every concept described here is implemented in src/models/enhanced_gpt.py.

3.1 — Tokenization (Text → Numbers)

Neural networks cannot process text directly — they only understand numbers. Tokenization is the process of converting raw text into a sequence of integer IDs.

VV-GPT3 uses Byte Pair Encoding (BPE) via OpenAI's tiktoken library, the same tokenizer used by GPT-2, GPT-3, and GPT-4. BPE works by:

1

Starting with individual bytes/characters as the base vocabulary.

2

Iteratively finding the most frequently occurring pair of adjacent tokens in the training corpus and merging them into a single new token.

3

Repeating until the vocabulary reaches a target size (50,257 tokens for GPT-2).

// Example: BPE tokenization
Input: "to be or not to be"
Tokens: [1462, 307, 393, 407, 284, 307]
// Each number maps to a sub-word: "to" → 1462, " be" → 307, " or" → 393, etc.
// Notice: common words become single tokens. Rare words get split into pieces.

This is vastly superior to character-level tokenization (V1), where "to be" would become [t, o, ' ', b, e] — 5 tokens instead of 2. BPE gives the model a much richer vocabulary and shorter sequences to process, dramatically improving both speed and output quality.

3.2 — Embeddings (Numbers → Vectors)

Raw token IDs (like 1462) are meaningless numbers. Embeddings convert each token ID into a dense vector (a list of floating-point numbers) that the network can actually learn from.

VV-GPT3 uses two embedding layers that are summed together:

Token Embedding (wte)

A lookup table of shape [50257 × n_embd]. Each of the 50,257 vocabulary tokens has its own learnable vector of dimension n_embd (e.g., 384 for Tiny). During training, these vectors adjust so that semantically similar tokens end up with similar vectors.

Position Embedding (wpe)

A lookup table of shape [block_size × n_embd]. Transformers process all tokens simultaneously (unlike RNNs which read left-to-right), so they have no inherent sense of word order. Position embeddings inject location information: token at position 0 gets one vector, position 1 gets another, etc.

// The combined embedding for each token
x = TokenEmbed(token_id) + PosEmbed(position)
x = Dropout(x)

// For the Tiny preset:
// TokenEmbed: [50257 × 384] — each token gets a 384-dim vector
// PosEmbed: [256 × 384] — each position (0-255) gets a 384-dim vector
// Result x: [batch × seq_len × 384]

3.3 — Causal Self-Attention (The Core Innovation)

Self-attention is the mechanism that allows the model to understand relationships between words. For every token in the sequence, attention asks: "Which other tokens in this sequence are most relevant to predicting what comes next?"

The process works by projecting each token's embedding into three separate vectors:

Query (Q)

"What am I looking for?"

Key (K)

"What do I contain?"

Value (V)

"What information do I carry?"

// Step 1: Project embeddings into Q, K, V
Q, K, V = Linear(x) // Single matrix multiply, then split into 3

// Step 2: Compute attention scores
scores = (Q × Kᵀ) / √d_k
// d_k = dimension per head (384/6 = 64 for Tiny)
// The √d_k scaling prevents scores from growing too large

// Step 3: Apply causal mask (THE critical constraint)
scores = scores + CausalMask
// CausalMask is a lower-triangular matrix where:
// allowed positions = 0
// future positions = -∞
// This forces the model to ONLY look at past/present tokens, never the future.
// Without this, the model would "cheat" by seeing the answer.

// Step 4: Normalize to probabilities and extract values
weights = Softmax(scores) // Each row sums to 1.0
output = weights × V // Weighted sum of value vectors

Multi-Head Attention: Instead of computing one set of Q/K/V, VV-GPT3 computes multiple independent "heads" in parallel (6 heads for Tiny, 16 for Large). Each head can learn to attend to different types of relationships — one head might learn syntax, another might learn co-reference, etc. The outputs are concatenated and projected back to the original dimension.

Why "Causal"? The causal mask is what makes this a language model rather than just a pattern detector. By preventing the network from seeing future tokens, we force it to learn to predict the next token given only the past — which is exactly what we want during text generation.

3.4 — Feed-Forward Network (MLP)

After attention, each token's representation passes through a feed-forward network (also called an MLP — Multi-Layer Perceptron). This is where the model does its "thinking" about what it's learned from the attention step.

// Two linear transformations with GELU activation
hidden = GELU(Linear₁(x)) // Expand: [384] → [1536] (4× expansion)
output = Linear₂(hidden) // Contract: [1536] → [384]
output = Dropout(output)

// GELU (Gaussian Error Linear Unit) is a smooth activation function
// Unlike ReLU which has a hard cutoff at 0, GELU smoothly gates values,
// allowing small negative values through. This leads to better training.

3.5 — The Transformer Block (Putting It Together)

A single Transformer Block combines attention and the FFN with residual connections and layer normalization. VV-GPT3 uses pre-normalization (normalizing before each sub-layer rather than after), which is more numerically stable during training.

// One Transformer Block (pre-norm architecture)
x = x + Attention(LayerNorm(x)) // Residual connection around attention
x = x + FFN(LayerNorm(x)) // Residual connection around FFN

// The residual connections (the "+ x" parts) are critical.
// They create "skip paths" that allow gradients to flow directly
// backward through the network without vanishing. Without them,
// deep networks (6+ layers) would be nearly impossible to train.

The full GPT model stacks multiple of these blocks sequentially. The Tiny preset uses 6 blocks. Each block refines the representation — early blocks tend to learn basic syntax, while later blocks learn more abstract semantic patterns.

3.6 — Output Projection & Cross-Entropy Loss

After passing through all transformer blocks, a final LayerNorm is applied, followed by a linear projection that maps the hidden state back to vocabulary size (50,257 dimensions). This produces logits — raw, unnormalized scores for every possible next token.

// Final output
x = LayerNorm(x) // Final normalization
logits = Linear(x) // [batch × seq_len × 50257]

// During training: compute loss
loss = CrossEntropy(logits, targets)

// Cross-entropy loss measures how "surprised" the model is by the
// correct answer. If the model assigns high probability to the correct
// next token, loss is LOW. If it's surprised, loss is HIGH.

// A perfect model: loss → 0 (never surprised)
// Random guessing with 50,257 tokens: loss ≈ 10.82 (ln(50257))
// A good small model on Shakespeare: loss ≈ 3.9-4.2

3.7 — Weight Tying

The token embedding matrix (which converts token IDs → vectors) and the output projection matrix (which converts vectors → token probabilities) have the exact same shape: [50257 × n_embd]. VV-GPT3 ties them together — they share the same physical memory and learn the same weights.

This has two benefits: it reduces the parameter count by ~20% (eliminating ~19M duplicate parameters in the Tiny model), and it creates a mathematical symmetry where the model's understanding of "what a token means" (embedding) is the same as its understanding of "when to output a token" (projection).

Model Architecture Presets

VV-GPT3 ships with 6 pre-configured model sizes. The key architectural dimensions are: number of transformer layers (depth), number of attention heads (parallel attention patterns), and embedding dimension (vector width).

Head Size = Embed Dim / Heads. FFN Size = 4 × Embed Dim (standard GPT-2 expansion factor). The actual parameter count depends on block_size and vocab_size, and is reduced by weight tying.

The Training Process

Training is the process of adjusting the millions of numerical parameters (weights) inside the network so that it becomes better at predicting the next token. Here is exactly what happens at each training step:

5.1 — The Training Loop

1

Sample a batch: Randomly pick batch_size (e.g., 64) chunks of block_size (e.g., 256) consecutive tokens from the training data. The input is tokens [0…254], and the target is tokens [1…255] (shifted by one).

2

Forward pass: Feed the input through the entire model (embeddings → transformer blocks → output projection) to produce logits for every position.

3

Compute loss: Compare the model's predictions (logits) against the actual next tokens (targets) using cross-entropy loss. This produces a single number measuring how wrong the model is.

4

Backward pass (Backpropagation): Compute the gradient of the loss with respect to every single parameter in the network. This tells us the direction and magnitude each parameter should be adjusted to reduce the loss.

5

Gradient clipping: If any gradient is excessively large (norm > 1.0), scale all gradients down. This prevents "exploding gradients" that can destabilize training.

6

Optimizer step (AdamW): Update all parameters using the AdamW optimizer, which maintains per-parameter momentum and adaptive learning rates, plus weight decay for regularization.

7

Repeat for the configured number of iterations (e.g., 2000–3000).

5.2 — Training Loss vs Validation Loss

The dataset is split 90/10 into training data and validation data. Training loss is computed on data the model learns from. Validation loss is computed on data the model has never seen.

Training loss tells you how well the model has memorized the training data. Validation loss tells you how well the model has generalized — learned patterns that apply to new text. The gap between these two numbers is the key diagnostic:

📈 Healthy Training

Both losses are dropping together. The gap between them is small. The model is learning real patterns.

📊 Plateau

Val loss has stopped improving. Training loss may still drop. The model has extracted most learnable patterns.

⚠️ Overfitting

Training loss drops but val loss rises. The model is memorizing exact sequences instead of learning patterns. Generated text will be repetitive.

5.3 — Overfitting Detection (The Tripwire)

VV-GPT3 includes an automatic overfitting tripwire. The backend monitors validation loss at every evaluation interval. If val_loss increases for 5 consecutive evaluations, the system fires a warning recommending that you terminate training.

Throughout training, the system saves two types of checkpoints: the latest checkpoint (most recent state) and the best checkpoint (lowest val_loss ever achieved). Even if you train past the optimal point, the best model is always preserved. This is the checkpoint used as your final model.

5.4 — Hyperparameter Guide

Parameter What It Controls Effect of Increasing
Learning Rate How large each weight update is Faster training but risks overshooting minima. Too high → unstable loss. Too low → slow convergence.
Iterations Total number of training steps More exposure to data. Diminishing returns after the model plateaus. Risk of overfitting if too many.
Block Size Context window (how many tokens the model can "see" at once) Better long-range coherence but quadratically more memory and compute. 256 is balanced; 512 for longer context.
Batch Size Number of text chunks processed per step Smoother gradient estimates, faster per-epoch, but more memory. 32–64 is typical.
Dropout Fraction of neurons randomly disabled per step More regularization — forces redundant learning, delays overfitting. 0.1 = light, 0.2 = aggressive.
Eval Interval Steps between validation checks Lower = more frequent monitoring but slower training (validation takes compute). 50 steps is standard.

Inference & Text Generation

Once trained, the model generates text by repeatedly predicting the next token, appending it to the sequence, and feeding it back in. This is called autoregressive generation.

6.1 — The Generation Loop

// Autoregressive generation with KV-Caching

prompt = tokenize("to be or not to be")
kv_cache = empty

for i in range(max_tokens):
  // First iteration: process entire prompt
  // Later iterations: process ONLY the last token (O(1) speed!)
  logits, kv_cache = model(last_token, kv_cache)

  // Apply temperature scaling
  logits = logits / temperature

  // Convert to probabilities
  probs = Softmax(logits)

  // Sample the next token from the probability distribution
  next_token = sample(probs)
  output.append(next_token)

6.2 — KV-Caching (Why Generation Is Fast)

Without caching, generating each new token would require re-processing the entire sequence from scratch — for a 400-token output, that's 400 full forward passes of increasing length. With KV-Caching, the model stores the Key and Value tensors from all previous positions in memory. When generating the next token, it only needs to compute attention for the single new token against the cached K/V pairs.

This reduces generation from O(n²) to O(n) total compute, making real-time chat possible even on consumer hardware.

6.3 — Temperature & Sampling

Temperature controls the randomness of the model's output by scaling the logits before applying softmax:

🧊 Low (0.1–0.4)

Dividing logits by a small number makes them larger, sharpening the probability distribution. The model almost always picks the highest-probability token. Output is deterministic and repetitive but "safe".

⚖️ Medium (0.6–0.8)

Balanced randomness. The model mostly picks high-probability tokens but occasionally surprises with creative choices. Good default for coherent yet varied output.

🔥 High (1.0–1.5)

Dividing by a large number flattens the distribution. Lower-probability tokens get a real chance. Output is creative and diverse but can become incoherent at extremes.

6.4 — What Gets Saved in a .pt Checkpoint (~350MB)

A .pt file is a serialized PyTorch state dictionary. At ~350MB for a Tiny model, it contains:

model — All ~30M learned weight parameters for Tiny (floats, ~100MB uncompressed)
optimizer — AdamW momentum states: 2 float buffers per parameter (~200MB)
model_args — Architecture config (layers, heads, embed dim, block size)
train_config — Training hyperparameters (LR, weight decay, betas)
processor — Tokenizer mapping (stoi/itos dictionaries for decoding)
training_metadata — Best val_loss, training/val loss curves, timing data
iter_num — Which training step this checkpoint was saved at

The optimizer state accounts for most of the file size — it stores two running averages (first and second moment) for every parameter, which is what allows AdamW to make intelligent per-parameter learning rate adjustments.

System Architecture & Data Flow

The platform is separated into a Python Backend (Flask + SocketIO) handling all computation, and a Vanilla HTML/CSS/JS Frontend providing the glassmorphic UI. Communication is real-time via WebSockets.

7.1 — Technology Stack

Python 3.9+ PyTorch 2.0+ Flask Flask-SocketIO Tiktoken Eventlet Chart.js Bootstrap 5

7.2 — Real-Time Telemetry Pipeline

During training, the backend runs the training loop in a background thread. At each evaluation interval, it emits SocketIO events to the browser:

Backend (web_app.py)
├─ emit_and_record('training_progress', {...}) // Loss, iteration, ETA
├─ emit_and_record('checkpoint_saved', {...}) // Best/latest saved
├─ emit_and_record('training_log', {...}) // Status messages
└─ training_event_history[] // Server-side buffer for page persistence

Frontend (train.html)
├─ socket.on('training_progress') → updateChart() + updateTelemetry()
├─ socket.on('checkpoint_saved') → updateBestLine() + addLogEntry()
├─ socket.on('training_log') → addLogEntry() + overfitDetect()
└─ fetch('/training_history') → // On page load: replay all events

The emit_and_record pattern ensures that every event is both pushed to connected browsers and stored in a server-side buffer (training_event_history). If you navigate away and return, the frontend fetches this history and replays all events, reconstructing the full UI state — logs, chart, telemetry, and overfit detection — with zero data loss.

7.3 — Chat Persistence

The inference chat page stores conversation history in the browser's localStorage, keyed by model name. Switching models loads that model's chat history. Page refreshes preserve the conversation. The active parameter bar (temperature, max tokens, model name) updates in real-time as you adjust settings.
